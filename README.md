# VideoTranscriber

Desktop-приложение на PySide6 для расшифровки видео в русский текст и сборки готового промта для LLM. Работает на CPU-only: `faster-whisper` запускается с `device="cpu"` и `compute_type="int8"`, а `ffmpeg` берется из `imageio-ffmpeg`.

## Что умеет

- Открывает папку с видео и рекурсивно находит `MP4`, `MKV`, `AVI`, `MOV`, `WEBM`.
- Сортирует видео от самого нового к старому и автоматически выбирает новый файл для обработки.
- Поддерживает drag&drop видеофайлов или папок в список.
- У каждого файла есть галочка `process file`: сейчас за один запуск обрабатывается только один отмеченный файл.
- Модели Whisper: `tiny`, `base`, `small`, `medium`, `large-v3-turbo`, `large-v3`.
- Показывает ориентиры по RAM, размеру модели, скорости и качеству.
- Настройки скорости: режим `Быстро / Баланс / Качество` и количество потоков CPU для faster-whisper.
- Фоновая обработка через `QThread`, интерфейс не зависает.
- Лог, прогресс текущего файла и таймер обработки.
- Редактируемый шаблон промта с переменной `{text}`. Дефолтный промт просит LLM вернуть валидный JSON со смысловыми блоками.
- Опция `Сжимать расшифровку для LLM`: убирает повторы и часть мусорных слов без деления текста на чанки.
- Автосохранение готового промта в `.txt`.
- Вкладка `Ответ LLM → Word`: вставьте ответ модели и сохраните его в `.docx`. Если ответ пришел JSON по шаблону, он будет превращен в читаемые разделы.

## Почему без собеседников

`faster-whisper` не определяет реальные голоса. Простая разметка по паузам выглядит убедительно, но часто ошибается и портит итоговый промт. Поэтому приложение оставляет чистую расшифровку без выдуманных `Собеседник 1/2`.

Настоящая diarization возможна через отдельные библиотеки вроде `pyannote.audio`, но это тяжелее, требует отдельные модели, часто HuggingFace token и заметно усложняет PyInstaller-сборку.

## RAM и модели

Оценки ниже приблизительные для CPU/int8. Реальное потребление зависит от длины аудио, ОС и версии библиотек.

| Модель | RAM | Размер кэша модели | Скорость на CPU | Качество русского |
| --- | ---: | ---: | --- | --- |
| `tiny` | 1-2 ГБ | ~75 МБ | Очень быстро | Черновик, часто плохо |
| `base` | 2 ГБ | ~150 МБ | Быстро | Часто слабовато |
| `small` | 3-4 ГБ | ~500 МБ | Нормально | Рекомендуемый баланс |
| `medium` | 6-8 ГБ | ~1.5 ГБ | Медленно | Лучше для сложной речи |
| `large-v3-turbo` | 8-10 ГБ | ~1.6 ГБ | Средне/медленно | Сильная модель |
| `large-v3` | 12-16 ГБ | ~3 ГБ | Очень медленно | Лучшее качество |

Для нормального русского лучше начинать с `small`. `tiny` и `base` годятся только для быстрых черновиков.

## Установка для запуска из исходников

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
python main.py
```

На Linux/macOS:

```bash
source .venv/bin/activate
```

## Сборка

Сборку нужно делать отдельно на каждой целевой ОС.

Windows:

```bash
pyinstaller --onefile --windowed --icon=icon.ico --name VideoTranscriber main.py
```

Linux:

```bash
pyinstaller --onefile --windowed --icon=icon.png --name VideoTranscriber main.py
```

macOS:

```bash
pyinstaller --windowed --icon=icon.png --name VideoTranscriber main.py
```

Результат появится в `dist/`: `VideoTranscriber.exe` на Windows, `VideoTranscriber` на Linux или `VideoTranscriber.app` на macOS.

## Использование

1. Нажмите `Открыть папку` и выберите папку с видео.
2. Сверху будет самый новый файл, он выбирается автоматически. При необходимости поставьте галочку у другого файла.
3. Выберите модель. Для русского обычно лучше `small` или выше.
4. Для длинного созвона поставьте режим `Быстро` и 70-90% доступных потоков CPU.
5. Оставьте включенным `Сжимать расшифровку для LLM`, если созвон длинный.
6. Оставьте включенным `Автосохранять промт в .txt` или выберите папку результата.
7. Нажмите `Запустить`.
8. После завершения скопируйте готовый промт и отправьте его в любую LLM.
9. Скопируйте ответ LLM, откройте вкладку `Ответ LLM → Word` и нажмите `Вставить из буфера`.
10. Нажмите `Сохранить Word`. Файл `.docx` появится в выбранной папке результата.

## Скорость Whisper на CPU

Для двухчасового созвона CPU-only обработка может идти долго, особенно на `medium` и `large`. Если диспетчер задач показывает около 20% CPU, обычно стоит увеличить `Потоки CPU` и включить режим `Быстро`.

- `Быстро`: `beam_size=1`, быстрее всего, качество немного ниже.
- `Баланс`: `beam_size=3`, хороший вариант по умолчанию.
- `Качество`: `beam_size=5`, медленнее, лучше для сложной речи.

Если компьютер начинает тормозить, уменьшите число потоков. Если качество в режиме `Быстро` плохое, попробуйте `Баланс` или модель выше, например `small` вместо `base`.

## Формат ответа LLM

Дефолтный промт просит модель вернуть JSON:

```json
{
  "title": "короткое название",
  "short_summary": "резюме",
  "blocks": [
    {
      "number": 1,
      "title": "название блока",
      "text": "главный текст блока",
      "decisions": [],
      "tasks": [{"who": null, "what": "что сделать", "deadline": null}],
      "open_questions": [],
      "risks": []
    }
  ],
  "global_tasks": [],
  "unclear_or_risky": []
}
```

JSON удобен тем, что приложение может автоматически разложить ответ по разделам в Word. Если модель вернет обычный текст или markdown, он тоже сохранится в `.docx`, просто без JSON-разбора.

## Длинные созвоны и размер входа

Опция `Сжимать расшифровку для LLM` делает только механическое сжатие: нормализует пробелы, убирает частые слова-паразиты и повторы подряд. Она не пересказывает смысл локально и не делит текст на чанки, поэтому модель не должна путать технические фрагменты со смысловыми главами.

Если созвон очень длинный и чат LLM все равно не принимает промт целиком, лучше отправить сохраненный `.txt` как файл в интерфейсе LLM. Если файловая загрузка недоступна, придется делить текст вручную уже вне приложения, но в обычном режиме приложение чанки в промт не добавляет.

## Модели и интернет

Модели скачиваются при первом выборе из HuggingFace в системный кэш, обычно `~/.cache/huggingface/`. Первый запуск конкретной модели требует интернет и может занять время. После скачивания модель используется из кэша.

## Частые проблемы

- Некорректный видеофайл: приложение пишет ошибку в лог и завершает текущий запуск.
- Модель не скачалась: проверьте интернет, доступ к HuggingFace и свободное место на диске.
- Нехватка памяти: выберите модель меньше, например `small`, `base` или `tiny`.
- Медленная обработка: CPU-only транскрипция больших видео может занимать много времени, особенно на старых процессорах.
